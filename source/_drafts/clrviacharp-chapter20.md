## 예외 처리의 구조
다음의 C# 코드는 예외 처리를 위한 일반적인 구조를 보여주고 있다. 이 코드는 예외 처리 블록이 어떤 모습을 띠고 있으며, 이를 통해 무엇을 할 수 있을지를 나타내고 있다.
~~~
private void SomeMethod() {
    try {
        // 안정적으로 오류 상황이 처리되어야 하고 깔끔하게 정리되어야 하는 코드를 여기에 둔다.
    } catch (InvalidOperationException) {
        // InvalidOperationException 발생 시 처리해야 하는 코드를 여기에 둔다.
    } catch (IOException) {
        // IOException 발생 시 처리해야 하는 코드를 여기에 둔다.
    } catch {
        throw;
    } finally {
        // try 블록을 벋어날 때 수행해야 하는 정리 코드를 여기에 둔다.
        // 이 블록 내의 코드는 예외의 발생 여부와 상관없이 항상 수행된다.
    }
    // finally 블록 아래쪽의 코드는 try 블록 내에서 예외가 발생하지 않은 경우 혹은 예외가 발생하였고
    // catch 블록을 통해서 예외를 잡았으나, 그 내부에서 다시 예외를 발생시키지 않은 경우에만 수행된다.

~~~
### try 블록
try 블록 내에 두은 코드는 정리 작업이 반드시 필요하거나 혹은 예외를 극복해야 할 필요가 있는 코드를 두게 된다. 이 외에도 try 블록 내에는 잠재적으로 예외를 발생시킬 가능성 있는 코드를 두기도 한다.
### catch 블록
catch 블록은 예외 발생에 대응하기 위한 코드를 포함한다.

catch 키워드 다음에는 괄호로 묶인 표현식이 나오는데 이를 **catch 타입**이라고 부른다. C#에서는 System.Exception을 상속한 타입만을 catch 타입으로 지정할 수 있다.

catch 블록의 마지막 부분에서는 다음에 열거한 세가지 작업 중 하나를 수행할 수 있다.
* 동일한 예외를 다시 던져서 호출 스택의 상위로 예외를 전달한다.
* 발생한 예외에 추가적인 정보를 더한 다른 타입의 예외를 던져서 호출 스택의 상위로 새로 생성한 예외를 전달한다.
* catch 블록을 벗어나서 이하의 코드를 수해앟도록 한다.
### finally 블록
finally 블록 내의 코드는 수행이 보장되기 때문에 try 블록 내에서 수해오디는 코드들이 추가적인 정리 작업을 필요로 하는 경우에 주로 사용된다.

## System.Exception 클래스
CLR 자체는 Int32로부터 String 혹은 어떤 타입이라도 예외로 발생시킬 수 있도록 허용하고 있다. 이와는 다르게 마이크로 소프트는 모든 프로그래밍 언어가 모든 타입의 에외를 발생시키는 것을 허용하지 않기로 결정했고, System.Exception 타입을 정의하고, CLS를 따르는 모든 프로그래밍 언어는 반드시 System.Exception 혹은 이를 상속한 타입의 예외만을 발생시킬 것을 규격화하였다. 그리고 System.Exception을 상속한 타입을 CLS 준수 예외(CLS-compliante exception)로 부르기로 결정하였다.

| 속성 | 접근 | 타입 | 설명 |
| :- | :- | :- | :- |
| Message | 읽기 전용 | String | 예외가 발생한 이유를 설명한 텍스트를 포함하고 있다. |
| Data | 읽기 전용 | IDictionary | 키-값 쌍의 컬렉션에 대한 참조를 담고 있다. 보통 예외를 발생시킨 코드에서 예외 발생 직전에 이 컬렉션에 항목들을 포함시킨다. 예외를 잡은 코드는 이 컬렉션 내의 항목들을 살펴서 예외 복구 과정에 활용할 수 있다. |
| Source | 읽기/쓰기 | String | 예외를 발생시킨 어셈블리의 이름을 가지고 있다 |
| StackTrace | 읽기 전용 | String | 예외가 발생된 원인을 찾기 위한 실마리를 제공하기 위하여 호출된 메서드를의 이름과 원형이 저장되어 있다. 디버깅 시에 사앙히 귀중한 정보가 된다. |
| TargetSite | 읽기 전용 | MethodBase | 예외를 발생시킨 메서드의 이름을 가지고 있다. |
| HelpLink | 읽기 전용 | String | 발생한 예외가 무엇인지 사용자가 이해할 수 있도록 제공되는 문서의 URL 정보를 가지고 있다. |
| Inner Exception | 읽기 전용 | Exception | 현재 발생한 예외가 다른 예외를 처리 중에 새롭게 발생한 예외인 경우, 이전에 발생한 예외를 담고 있다. 읽기 전용이며, 보통 null이다. public GetbaseException 메서드를 이용하면 Inner Exception으로 이어진 연결리스트를 타고 들어가 가장 최초에 발생한 예외가 무엇인지 얻을 수 있다. |
| HResult | 읽기/쓰기 | Int32 | 이 32비트 값은 관리 코드와 비관리 코드의 경계를 넘나들때 사용되는 값이다. |

예외가 발생하면 CLR은 내부적으로 어느 위치에서 throw 명령을 수행하였는지를 기록해둔다. catch 블록이 예외를 잡는 경우에도 CLR은 예외를 잡은 위치를 기록해둔다. catch 블록 내에서 예외의 StackTrace 속성에 접근을 시도하면 속성 읽기를 위해 구현된 코드가 CLR을 호출하고 CLR은 예외가 발생했던 위치로부터 예외를 잡은 위치 사이에 있는 모든 메서드들을 문자열로 작성해둔다.

> **중요** 예외를 발생시키면, CLR은 예외 발생위치를 재설정한다. 이 말인 즉 CLR은 가장 마지막 예외가 발생한 지점 하나만을 기억하고 있다는 것이다.

## 예외 발생시키기
예외를 발생시키도록 메서드를 구현하기로 결정하였다면 다음 두 가지 사항을 신중하게 생각하고 검토해봐야 한다.

첫 번째는 Exception 계통의 예외 중 어떤 타입을 예외로 발생시킬 것인가 하는 것이다 당연한 이야기지만 예외 상황에 가장 잘 부합하는 예외를 발생시키고 싶을 것이다. 안정적으로 예외를 처리하는 코드를 작성하려면 호출 스택을 거슬러 올라가더라도 실패한 메서드가 무엇인지 정확히 알 수 있어야 한다.

두 번째는 예외 타입의 생성자에 어떤 문자열 메시지를 전달할지를 결정하는 것이다. 예외를 발생시킬 때, 메서드가 작업을 완료하지 못한 이유를 자세히 설명하는 문자열 메세지를 포함하는 것이 좋다.

## 지침과 모범 사례
### finally 블록을 아낌없이 사용하라
스레드가 어떤 예외를 발생시키더라도 finally 블록은 항상 수행될 것임을 보장 받을 수 있기 때문이다. 다음에 finally 블록을 이용하여 정리 코드를 어떻게 작성하는지 그 예를 나타내었다.
~~~
public sealed class SomeType() {
    private voide SomeMethod() {
        FileStream fs = new FileStream("@C:\Data.bin ", FileMode.Open);
        try {
            Console.WriteLine(100 / fs.ReadByte());
        }
        finally {
            // 예외 발생(예: 첫 번째 바이트가 0이라면) 여부와 상관 없이
            // 파일 닫기를 수행할 수 있도록 정리 코드를 finally 블록에 둔다.
            if (fs != null)  fs.Dispose();
        }
    }
}
~~~
정리 코드를 수행하는 것은 정말 중요한 작업이므로, 많은 프로그래밍 언어에서 이러한 작업을 쉽게 수행할 수 있는 방법들을 제공하고 있다. 예를 들어 C# 언어에서는 lock, using foreach 등을 사용할 경우 자동적으로 try/finally 블록을 생성한다. 또한 C# 컴파일러는 객체가 Finalize 메서드를 오버라이드할 경우에 자동적으로 try/finally 블록을 생성한다. using 문장을 사용하면 사용자가 작성한 코드를 try 블록 내부로 옮기고 finally 블록에 정리 코드를 포함시킨다. 각 키워드별 세부 내용은 다음과 같다.
* lock 문장을 사용하면, finally 블록 내부에 잠금 해제 코드를 포함시킨다.
* using 문장을 사용하면, finally 블록 내부에 객체의 Dispose 메서드를 호출하는 코드를 포함시킨다.
* foreach 문장을 사용하면, finally 블록 내부에 IEnumerator가 참조하는 객체의 Dispose 메서드를 호출하는 코드를 포함시킨다.
* 파괴자를 정의하면, finally 블록 내부에서 기본 클래스의 Finalize 메서드를 호출한다.

### 모든 예외를 잡으려 하지 마라
예외를 어떻게 사용하는 것이 좋은지에 대해서 올바른 교육을 받지 못한 개발자들이 정말 광범위하게 저지르고 있는 실수 중에 하나가 바로 너무 자주 catch 블록을 사용한다는 것과, 올바르지 않게 catch 블록을 사용한다는 것이다.

### 예외를 안정적으로 극복하기
### 복구할 수 없는 예외 발생 시 부분적으로 완료된 작업들을 취소하여 상태 유지
보통 하나의 메서드는 단일의 추상화된 작업을 수행하는 여러 메서드를 호출하도록 작성된다. 그런데 이렇듯 내부적으로 사용된 여러 메서드들 중 일부는 성공적으로 작업을 완료하였지만 또 다른 일부는 그렇지 않을 수 있다. 예를 들어 여러 객체들을 디스크에 파일로 저장하려 한다고 가정해보자. 10개 즈음을 저장하고 났더니 예외가 발생되었다고 해보자. 이 제 발생된 예외는 메서드의 호출자에게 전달될 수 있을 것이다. 하지만 파일의 상태는 어떻게 되었을까? 파일은 저장하고자 했던 객체의 일부만을 저장할 수 있었으므로 손상된 상태가 된다. 만일 응용프로그램이 부분적으로 완료된 작업들을 취소할 수 있어서, 파일의 상태가 일부 객체를 저장하기 이전 상태로 돌아갈 수 있다면 훌륭할 것 같다. 다음에 이러한 방법을 구현하는 예를 나타냈다.
~~~
public void SerializeObjectGraph(FileStream fs, IFormatter formatter, Object rootObj) {
    // 파일에서의 현재 위치를 저장해둔다.
    Int64 beforeSerialization = fs.Position;

    try {
        // 파일에 객체 그래프를 저장(serialize)한다.
        formatter.Serialize(fs, rootObj);
    }
    catch {
        // 만일 조금이라도 잘못된 부분이 있으면, 파일을 이전 상태로 재설정한다.
        fs.Position = beforeSerialization;

        // 파일 크기 재설정
        fs.SetLength(fs.Position);

        // 참고: 파일 저장이 실패하였을 때만 스트림을 재설정해야 하기 때문에
        // 앞쪽의 코드는 finally 블록에 둘 수 없다.

        // 호출자에게 예외가 발생하였음을 알려주기 위해서 동일한 예외를 다시 던진다.
        throw;
    }
}
~~~

### 계약을 유지하기 위해서 세부 구현사항 숨기기
일부 상황에서 특정 타입의 예외를 잡더라도 그와는 다른 타입의 예외를 다시 던지는 것이 효과적일 때가 있다. 이렇게 하는 유일한 이유는 메서드의 계약사항을 유지하기 위해서이며, 새롭게 생성하는 예외는 대체로 더욱 구체적인 예외 타입이어야 한다. 이름으로 전화번호를 찾는 메서드를 가지고 있는 PhoneBook이라는 타입을 어떻게 작성할지 생각해 보자. 다음에 의사 코드를 나타내보았다.
~~~
internal sealed class PhoneBook {
    private String m_pathname;  // 주소책의 파일 위치 지정

    public String GetPhoneNumber(String name) {
        String phone;
        FileStream fs = null;
        try {
            fs = new FileStream(m_pathname, FileMode.Open);
            // 지정한 이름을 찾을 때까지 fs로부터 내용을 읽어 오는 코드를 여기에 둔다.
            phone = /* 찾는 전화번호 */
        }
        catch (FileNotFoundException e) {
            // 찾고자 하는 이름을 포하ㅁ하고, 원래 발생한 예외를 내부 예외로 취하는
            // 다른 형태의 예외를 발생시킨다.
            throw new NameNotFoundException(name, e);
        }
        catch (IOEception e) {
            // 찾고자 하는 이름을 포하ㅁ하고, 원래 발생한 예외를 내부 예외로 취하는
            // 다른 형태의 예외를 발생시킨다.
            throw new NameNotFoundException(name, e);
        }
        finally {
            if (fs != null) fs.Close();
        }
        return phone;
    }
}
~~~
이러한 기법을 사용할 때에는 예외 발생 상황을 정확히 이해하고 있는 아주 구체적인 예외를 잡아야 한다. 또한 잡고자 하는 예외 타입을 상속하는 다른 예외 타입에 대해서도 알고 있어야 한다.

## 처리되지 않은 예외
예외가 발생하면, CLR은 발생한 예외의 타입에 부합하는 catch 블록을 찾기 위해서 스택을 거슬러 올라간다. 만일 예외 타입에 부합하는 catch 블록을 찾지 못하면, **처리되지 않은 예외**(unhandled exception)가 발생한다. CLR은 프로세스 내의 어떤 스레드에서라도 처리되지 않은 예외가 발생하면 프로세스를 종료시켜버린다.
