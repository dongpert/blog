### 소스 코드를 관리 모듈로 컴파일하기
공용 언어 런타임(CLR)은 이름에서 알 수 있듯이, 서로 다른 프로그램밍 언어들 사이에서 공동으로 사용할 수 있는 실행 환경(Runtime)이다. CLR의 핵심 기능들(예를 들어 메모리 관ㄹ, 어셈블리 로딩, 보안, 예외 처리, 스레드 동기화 등)은 CLR을 지원하는 어떠한 프로그래밍 언어라도 제약 없이 자유롭게 사용할 수 있다.

CLR을 지원하는 프로그래밍 언어라면 여러분이 원하는 대로 소스 코드를 작성할 수 있다. 그 다음, 해당 언어를 지원하는 컴파일러를 사용하여 문법을 점검하고 소스 코드를 분석하게 된다. 어떠한 컴파일러를 사용하든 최종적으로 **관리 모듈**(managed module)을 결과물로 얻게 된다.

네이티브 코드 컴파일러는 특정 CPU 아키텍처, 즉 x86, x64, ARM 아키텍처에 맞춘 코드를 생성한다. 모든 CLR 호환 컴파일러들은 네이티브 코드 컴파일러들과는 달리 IL 코드를 대신 만든다. IL 코드는 종종 **관리 코드**(Managed Code)로 불리는데 CLR이 실행을 관리하기 때문이다. 더 나아가서 IL을 만들 때, CLR을 대상으로 하는 모든 컴파일러들은 관리 모듈에 대한 전체 **메타데이터**를 작성하도록 해야 한다. 요약하면, 메타데이터는 모듈 내의 타입과 그들의 구성 멤버와 같은 정보들의 데이터 테이블의 집합이다. 더 나아가서 메타데이터는 내부가 아닌 외부에서 참조하는 모듈에 관한 정보, 예를 들어 외부에서 가져온 타입과 멤버들에 대한 정보들도 같이 포함하고 있다.

### 관리 모듈들을 하나의 어셈블리로 통합하기
CLR은 실제로 모듈들을 다루지는 않으며, 어셈블리를 다루게 된다. **어셈블리**는 한 번에 이해하기에는 다소 어려운 추상적인 개념이다. 첫 번째, 하나의 어셈블리는 하나 이상의 모듈이나 리소스 파일들에 대한 논리적인 그룹이다. 두 번째, 하나의 어셈블리는 재사용, 보안, 버전 관리의 가장 작은 단위다. CLR 환경에서는 하나의 어셈블리를 구성요소(Component)라고 부른다.

기본적으로, 컴파일러들은 만들어낸 관리 모듈을 하나의 어셈블리로 바꾸는 작업을 실행하는데, 다시 말하자면, C# 컴파일러는 매니페스트를 포함하는 관리 모듈을 만들어낸다는 것이다. 이 매니페스트는 어셈블리가 하나의 파일로 구성된다.

### 공용 언어 런타임 로딩
여러분이 만드는 어셈블리는 실행 가능한 응용프로그램 또는 DLL이든 관계 없이 실행 가능한 응용프로그램에서 사용할 것을 위하여 여러 타입들의 집합을 포함하고 있다. 당연히 CLR은 어셈블리 안에 들어있는 코드의 실행을 관리할 의무가 있다. 그래서 .NET Framework가 코드를 실행할 호스트 컴퓨터에 같이 배포되어야 한다.

%SYSTEMROOT%\SYSTEM32 디렉터리에 들어있는 MSCOREE.DLL 파일을 찾아보는 것으로 .NET Framework의 설치 여부를 알 수 있다. 이 파일이 존재한다는 것은 .NET Framework가 이미 설치된 것이다. 하지만 여러 버전의 .NET Framework가 동시에 같은 컴퓨터에 설치되어 있을 수도 있다. 만약 정확히 어떤 버전의 .NET Framework가 설치되어 있는지 알아내기를 원한다면 다음의 두 디렉터리 아래에 있는 하위 디렉터리들을 확인해 보면 된다.
~~~
%SystemRoot%\Microsoft.NET\Framework
%SystemRoot%\Microsoft.NET\Framework64
~~~

### 어셈블리 코드 실행하기
관리 어셈블리는 메타데이터와 IL을 같이 포함한다고 하였다. IL은 CPU에 독립적인 기계어 코드로 Microsoft가 외부의 몇몇 상용 및 학술용 언어/컴파일러 제작자들과의 상의하에 만든 것이다. IL은 대다수의 CPU 기계어보다 더욱 고차원의 언어다. IL은 객체의 타입을 다루며, 객체를 생성하고 초기화하거나, 객체의 가상 메서드를 호출하거나, 배열 요소들을 직접 관리하는 명령어들(Instructions)을 가지고 있다. 뿐만 아니라 예외 처리를 위한 Throw와 Catch에 대한 명령어도 가지고 있다. 그래서 IL을 객체 지향형 기계 언어(Object Oriented Machine Language)로 볼 수도 있다.

메서드를 실행하기 위하여, IL은 네이티브 CPU 명령어로 변환된다. 이 작업은 CLR의 JIT(Just-in-time) 컴파일러에 의하여 실행된다.

IL이 "실행시간"(Just-in-time)에 컴파일되므로 이 작업을 실행하는 컴파일러를 JIT **JIT 컴파일러** 혹은 **Jitter**로 부르기도 한다.

성능 저하(Performance Hit)는 메서드가 최초로 호출될 때에만 발생한다. 그 이후 발생하는 모든 후속 메서드 호출들은 네이티브 코드로서 빠른 속도로 실행되는데, 네이티브 코드에 대해서는 다시 내용을 검사하거나 컴파일해야 할 필요가 없기 때문이다.
