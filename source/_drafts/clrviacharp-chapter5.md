### 프로그래밍 언어의 기본 타입
컴파일러가 직접 지원하는 데이터 타입들을 **기본 타입**(Primitive Type)이라고 부른다. 기본 타입들은 프레임워크 클래스 라이브러리상에 정의된 타입들과 직접 연결된다. 예를 들어 C#의 경우 int 타입은 System.Int32 타입과 바로 연결된다. 이 때문에 네 줄의 코드는 예외 없이 모두 동일한 IL 코드를 만들어 내게 된다.
~~~
int a = 0;                              // 가장 쓰기 쉬운 문법이다.
SYstem.Int32 a = 0;                     // 비교적 쉽다.
int a = new int();                      // 불편하다.
System.Int32 a = new System.Int32();    // 명시적이지만 너무 돌아가는 문법이다.
~~~

개발자들이 완전한 타입 이름을 사용하도록 했으면 하는 바람이 있는데 이에 대한 이유는 다음과 같다.
* 많은 개발자들이 string 키워드와 String 타입을 각각 코드에서 어디에 어떻게 써야 하는지를 몰라 혼동하는 것을 봤따. 둘 사이에는 전혀 차이점이 없기 때문이다. 비슷한 이유로, int 타입이 32비트 운영체제에서는 32비트 정수로, 64비트 운영체제에서는 64비트 정수로 취급된다고 이야기하는 경우도 있다. 그런데 이 이야기는 완전히 잘못된 내용이며, C#에서 int는 오로지 System.Int32 타입으로만 연결되는, 운영체제의 종류에 무관하게 언제나 32비트형의 부호 있는 정수많은 다루는 타입이기 때문이다. 만약 Int32 타입을 사용하여 코드를 작성하게 한다면 개발자가 이런 부분에 있어서 혼동할 일이 없을 것이다.
* C#에서 long 키워드는 System.Int64 타입으로 연결되지만, 다른 프로그래밍 언어들 중에서는 long 키워드로 정의되는 타입이 16비트 부호 있는 정수 또는 32비트 부호 있는 정수로 연결되는 경우가 있다. 이러한 프로그래밍 언어를 사용하는 사람이 long 키워드를 사용하느 C# 코드를 읽을 때 잘못 이해할 가능성이 매우 크다.
* FCL에 들어있는 메서드들의 이름 중 상당수는 타입의 이름을 차용하는 것들이 존재한다. 예를 들어 BinaryReader 타입의 경우 ReadBoolean, ReadInt32, ReadSingle과 같은 메서드들이 존재하고, System.Convert 타입의 경우 ToBoolean, ToInt32, ToSingle 등의 메서드들이 존재한다. 물론 다음의 코드들은 지극히 정상적으로 처리되지만, 보기에는 매끄러워 보이지 않는다.
~~~
BinaryReader br = new BinaryReader(...);
float val = br.ReadSingle();    // 이상은 없지만 보기에 어색하다.
Single val = br.ReadSingle();   // 보기에 훨씬 자연스럽다.
~~~

C# 컴파일러는 타입 변환, 리터럴, 그리고 연산자에 대해서는 다음에서 보일 코드와 같은 기능들을 지원해준다.

첬째, 컴파일러는 다음의 예시에서 설명하는 타입들에 대해 암묵적이든 명시적이든 형 변환을 할 수 있도록 해줄 것이다.
~~~
Int32 i = 5;
Int64 l = i;
Single s = i;
Byte b = (Byte) i;
Int16 v = (Int16) s;
~~~
C# 은 암묵적 타입 변환을 "안전하다"고 판단하는 타입 변환 연산에 대해서만 허용하며, 이 동작은 Int32에서 Int64로의 변환과 같이 타입 변환 과정에서 데이터 손실이 없다고 판단할 때에만 적용된다. 하지만 잠재적으로 데이터 손실 가능성이 우려되는 타입 변환 연산에 대해서는 명시적인 타입 변환만을 허용한다.

### 기본 타입 연산의 오버플로우 여부 검사
C#은 개발자가 오버플로우를 어떻게 처리할 것인지에 대해 직접 선택할 수 있도록 하고 있따. 기본적으로, 오버플로우에 대한 검사는 사용하지 않도록 되어 있다.

C# 컴파일러가 오버플로우 검사를 수행할 수 있또록 IL 코드를 만들게 하기 위해서는 컴파일러 스위치 중 /checked+를 지정해야 한다. 만약 오버플로우가 발생하는 경우 System.OverflowException 예외를 발생시킨다.

더 나아가서 전역적으로 오버플로우 검사를 켜거나 끄는 것 말고도, 개발자가 필요에 의하여 지역적으로 특정 코드 영역에서만 오버플로우 검사를 켜거나 끄는 방법도 사용할 수 있다. C#은 checked와 unchecked 연산자를 사용하여 이를 유연하게 지원하고 있다.
~~~
byte b = 100;
b = checked((Byte) (b + 200));  // OverflowException이 발생한다.

checked {
    Byte b = 100;
    b = (Byte) (b + 200);
}
~~~

> **중요** : checked 연산자나 checked 코드 블록을 사용하는 것은 단순히 컴파일 시점에 만들어내는 IL 코드가 오버플로우 검사를 수행할 것인지 아닌지에 대한 것만을 결정하는 것에 불과하므로 다른 메서드를 호출하는 경우 메서드 내에서 발생하는 오버플로우에 대한 검사는 메서드가 어떻게 컴파일이 되었는지에 따라서 실행이 될 수도 되지 않을 수도 있다.
~~~
checked {
    // SomeMethod 메서드 안에서 byte형 변수에 400을 대입하려고 한다고 가정하겠다.
    SomeMethod(400);
    // SomeMethod가 checked 연산자나 코드 블록으로 컴파일되었는지의 여부에 따라서
    // OverflowException이 발생하거나 발생하지 않을 수 있다.
}
~~~

예기치 않은 결과가 나타나는 경우를 그 동안 필자는 많이 보았다. 대개는 사용자의 입력에 문자가 있는 경우이지만, 개발자가 예상하지 못한 시스템의 어떤 기능 때문에 발생하는 일이기도 했다. 그리고 이런 문제를 극복하기 위해 개발자들이 고려해야 할 만한 내용들을 정리해보려고 한다.
* UInt32나 UInt64와 같은 부호 없는 데이터 타입 대신 Int32나 Int64와 같이 부호 있는 데이터 타입을 가능한 사용할 것을 권한다. 이렇게 하면 컴파일러가 오버플로우나 언더플로우에 대한 검사를 더 잘 수행할 수 있다. 더 나아가서 Array나 String의 Length 속성과 같이 클래스 라이브러리의 여러 부분들은 부호 있는 값을 하드코딩된 형태로 반환하고 있으며, 되도록 이러한 값을 여러분의 코드에서 활용할 때는 캐스팅을 하지 않도록 만드는 것이 좋다. 캐스팅을 적게 할수록 코드를 알아보기 쉽고 관리가 용이해진다. 그리고 부호 없는 숫자 타입은 애초에 CLS 호환 사양이 안니다.
* 코드를 작성할 때 명시적으로 checked 연산자나 checked 블록을 사용하여 사용자가 입력한 데이터로 인하여 예기치 않은 오버플로우가 발생할 가능성이 있는 부분, 즉 사용자의 입력이나 클라이언트 컴퓨터 입력을 처리하는 부분에서 검증을 하도록 하는 것이 좋다.
* 코드를 작성할 때 오버플로우 검사를 하지 않아도 되는 부분, 예를 들어 해시 값을 계산하거나 체크섬 값을 계산하는 등의 작업에서는 unchecked 연산자나 unchecked 블록을 명시적으로 사용하도록 한다.
* checked나 unchecked 키워드를 사용하지 않은 모든 코드에 대해서는 기본적으로 오버플로우에 대한 검사를 직접 수행하고 필요한 경우 예외를 발생시키도록 만드는 것이 필요하다.

> **중요** : System.Decimal 타입은 매우 특별한 타입이다. C#과 Visual Basic .NET을 포함하여 많은 프로그래밍 언어들이 Decimal을 기본 타입처럼 취급하지만 CLR은 그렇지 않다. 달리 말하면, CLR이 Decimal 타입의 값을 다룰 수 있는 IL 명령을 제공하지 않는다. .Net Framework SDK 문서를 보면, Decimal 타입에 대해 Add, Subtract, Multiply, Divide 등의 연산을 위한 정적 메서드가 따로 존재한다는 것을 알 수 있다. 뿐만아니라 Decimal 타입에는 +,- -, *, / 등의 계산을 위한 연산자 오버로딩도 추가되어 있는 것을 볼 수 있다. 

> Decimal 타입을 사용하는 코드를 컴파일하면, 컴파일러는 Decimal에 대해 서술된 모든 계산식이 Decimal 타입의 멤버에 대한 호출로 대치된다. 이로 인해 Decimal 타입을 다루는 코드가 CLR의 기본 타입을 다루는 코드보다 느리게 동작하게 된다. 또한 Decimal 타입을 직접 다루는 IL 코드가 없기 때문에, checked와 unchecked 연산자 및 코드 블록, 그리고 컴파일러 스위치 모두 Decimal 타입에 대해서는 검사를 지원하지 않는다.

### 참조 타입과 값 타입
CLR은 참조 타입(Reference Type)과 값 타입(Value Type)이라는 두 종류의 타입 체계를 지원한다. 참조 타입은 관리되는 힙에 항상 할당되며, C#의 new 연산자가 만들어진 객체의 주소를 반환한다. 참조 타입을 사용할 때에는 반드시 명심해야 할 성능상의 고려사항들이 있는데, 그 중 우선 다음의 사항들을 유념해야 한다.
* 메모리는 반드시 관리되는 힙에 할당되어야만 한다.
* 힙에 할당되는 각 객체들은 코드상에 선언된 사항들 이외에 추가적인 멤버들을 포함하고 있으며 이들에 대한 초기화가 반드시 이루어지게 된다.
* 객체에 선언되는 필드들에 관한 메모리 영역은 반드시 모두 0으로 초기화된다.
* 관리되는 힙으로부터 객체를 할당하면서 가비지 컬렉션이 발생할 수 있다.

다른 객체를 가리키는 참조가 없는 인스턴스가 있을 경우 이 인스턴스의 타입이 값 타입으로 선언되어 있을 경우 이 변수의 인스턴스 자체가 곧 하나의 필드다. 변수 자체가 인스턴스의 필드이기 때문에, 필드를 얻어내기 위해서 역추적을 할 필요가 전혀 없다. 값 타입의 인스턴스는 가비지 컬렉터의 관리 대상이 아니기 때문에, 관리되는 힙을 덜 사용할 수 있고 응용프로그램의 생명주기 동안 발생하는 가비지 컬렉션의 횟수를 줄일 수 있다.

.NET Framework SDK 문서를 보면 어떤 타입이 참조 타입인지 값 타입인지를 분명하게 구분하고 있다. 문서 안에서 어떤 타입을 찾을 때, 참조 타입으로 선언되어있는 경우 클래스라고 표현하고 있다. 반면, 구조체나 열거 타입이라고 표현하는 것은 모두 값 타입이다. 

여러분만의 고유한 값 타입을 만들기 위하여 상속 받을 타입을 선택할 수는 없지만, 대신 값 타입은 기본적으로 하나 이상의 인터페이스를 구현할 수 있도록 해준다. 더 나아가서 모든 값 타입들은 봉인된(Sealed) 타입으로, 다른 타입의 기본 타입이 될수 없다.

다음의 코드는 참조 타입과 값 타입이 어떻게 다른지를 보여준다.
~~~
class SomeRef { public Int32 x; }
struct SomeVal { public Int32 x; }

static void ValueTypeDemo() {
    SomeRef r1 = new SomeRef();     // 힙에 할당된다.
    SomeVal v1 = new SomeVal();     // 스택에 할당된다.
    r1.x = 5;                       // 포인터 역참조 발생
    v1.x = 5;                       // 스택에서 변경됨
    Console.WriteLine(r1.x);      
    Console.WriteLine(v1.x);
    SomeRef r2 = r1;                // 참조(포인터)만을 복사한다.
    SomeVal v2 = v1;                // 스택에 메모리 공간을 할당하고 모든 필드를 복사한다.
    r1.x = 8;    
    v1.x = 9;
    Console.WriteLine(r1.x);        // "8"을 표시한다.
    Console.WriteLine(r2.x);        // "8"을 표시한다.
    Console.WriteLine(v1.x);        // "9"을 표시한다.
    Console.WriteLine(v2.x);        // "5"을 표시한다.
}
~~~

여러분이 사용할 타입을 설계할 때에는 새로 만드는 타입이 값 타입 또는 참조 타입 중 어떤 것으로 택하는 것이 좋을지 세심하게 고려해야 한다. 특정한 상황에서 값 타입은 참조 타입보다 더 나은 성능을 제공한다. 다음의 질문에 대한 대답이 모두 "예"인 경우, 새로 만드는 타입은 반드시 값 타입으로 선언해야 한다.
* 새로 만드는 타입이 기본 타입처럼 쓰인다. 구체적으로, 인스턴스 필드에서 수정할 내용이 없거나 수정할 수 없는 상태의 단순한 타입이다. 이렇게 변경해야 할 대상이 없거나 변경할 수 없도록 보호된 멤버들만을 포함하는 변경되지 않는 타입(Immutable Type)을 만들었다고 이야기한다. 사실 대부분의 값 타입들의 대해서 소유하는 필드를 모두 readonly로 설정하는 것을 권한다
* 다른 타입으로부터 상속 받을 필요가 없는 타입이다.
* 다른 타입의 기본 타입으로 쓰이게 할 계획이 없다.

인스턴스의 크기 또한 선택 기준에 있어서 큰 지표가 되는데, 기본적으로 값 타입으로 선언된 인스턴스를 함수의 매개변수로 지정할 경우 매번 해당되는 크기만큼 메모리 할당과 복제가 발생하기 때문에 이로 인해 성능에 또 다른 영향을 줄 수 있기 때문이다. 또한, 메서드가 값 타입의 객체를 반환할 때에는 메서드 호출자에 의하여 앞의 경우와 마찬가지로 해당 타입 크기만큼의 메모리공간을 할당하고, 메모리를 모두 복사하기 때문에 성능에 영향을 줄 수 있다.
* 인스턴스의 전체 크기가 약 16바이트 이내로 매우 작다.
* 인스턴스의 크기가 16바이트보다 크지만 메서드의 매개변수나 반환 타입으로 쓰이는 일이 없음을 확실히 할 수 있다.

값 타입의 주요한 이점은 관리되는 힙에 할당되는 객체가 아니라는 점이다. 물론 값 타입 역시 참조 타입과 비교했을 때 고유한 한계가 존재한다. 다음은 값 타입과 참조 타입이 다른 점을 구체적으로 서술한 것이다.
* 값 타입의 객체는 두 가지 형태로 표현되는데 박싱되지 않은 형태와 박싱된 형태이다.
* 값 타입은 System.ValueType 타입으로부터 상속 된다. 이 타입은 System.Object타입과 동일한 기본 메서드들을 제공한다.
* 값 타입을 기본 타입으로 사용하여 새로운 값 타입이나 참조 타입을 정의할 수 없으므로, 값 타입에서 가상 함수를 만들 수 없다. 값 타입에 오는 메서드는 추상 메서드일 수 없으며, 모든 메서드는 암묵적으로 재정의가 불가능하도록 선언된 상태가 된다.
* 참조 타입의 변수는 객체의 메모리 주소가 힙을 가리키도록 되어있다. 기본적으로 참조 타입의 변수가 만들어지면, 처음에는 null로 초기화되어 해당 변수가 어떠한 객체도 가리키지 않는 상태가 된다. 이 상태에서 변수에 대한 엑세스를 시도하면 NullReferenceException 예외가 발생할 것이다. 그러나 값 타입의 경우 타입 내부에 일정한 값을 항상 가지고 있고, 값 타입의 모든 필드 멤버들은 처음에 0으로 초기화 된다. 값 타입의 변수는 포인터가 아니기 때문에, 값타입에 대한 접근을 할 때 NullReferenceException 예외는 결코 발생하지 않는다.
* 값 타입의 변수를 다른 값 타입의 변수로 대입하려고 할 때 필드 단위로 하나씩 복제가 이루어지게 된다. 그러나 참조 타입의 변수끼리 대입이 발생하면 단순히 메모리 주소만 복제된다.

### 박싱된 값 타입과 박싱되지 않은 값 타입
값 타입은 참조 타입보다 가벼운에 그 이유는 관리되는 힙에 객체로 할당되지 않기 때문이라고 하였으며, 가비지 컬렉터가 관리하지 않으며 포인터로 지칭되지 않는다. 하지만 값 타입에 대한 참조를 얻어야만 하는 일이 자주 있다. 예를 들어 System.Collections 네임스페이스의 ArrayList 객체를 만들어서 Point 구조체를 담는 자료 구조 인스턴스를 형성했다고 가정해보자.
~~~
struct Point {
    public Int32 x, y;
}

public sealed class Program {
    public static void Main() {
        ArrayList a = new ArrayList();
        Point p;        // 힙 밖에 Point 객체를 할당한다.
        for (Int32 i = 0; i < 10; i++) {
            p.x = p.y = 1;
            a.Add(p);
        }
    }
}
~~~
반복문을 반복할 때마다 Point의 값 타입의 필드들이 초기화될 것이다. 그 다음, Point 타입의 인스턴스가 ArrayList 객체 안에 등록될 것이다. 하지만 잠시만 생각해보자. 실제로 ArrayList안에 저장되는 것은 과연 무엇일까? 이에 대한 답을 얻기 위해 ArrayList 타입의 Add 메서드를 살펴보면서 어떤 타입의 매개변수를 받도록 하고 있는지를 살펴봐야 한다.
~~~
public virtual Int32 Add(Object value);
~~~
이를 통하여 Add 메서드가 단순히 Object형의 매개변수를 받고 있으며, Add가 객체의 참조 내지는 포인터 주소를 요구한다는 것을 알 수 있으며 이 참조는 관리되는 힙상의 객체에 대한 주소가 된다는 것을 알 수 있다. 하지만 앞의 코드에서 p라는 Point 구조체, 즉 값 타입을 매개변수로 전달하였다. 이 코드가 작동하기 위해서는 Point 값 타입이 반드시 실제의 참조 타입으로 변환되어야 하고, 이 객체의 참조가 전달되어야 한다.

박싱이라는 작업을 통해서 값 타입을 참조 타입으로 변환하는 메커니즘이 있기에 가능하다. 내부적으로, 박싱이 일어나면서 다음과 같은 일이 일어나게 된다.
1. 관리되는 힙에 메모리가 할당된다. 이때 소요되는 메모리의 크기는 값 타입 내에 들어있는 필드들의 메모리 크기에 더하여 관리되는 힙에 할당되는 모든 객체들처럼 타입 객체 포인터와 동기화 블록 인덱스라는 필드를 포함하여 할당된다.
2. 값 타입의 필드들은 새로 할당된 힙 메모리에 복사된다.
3. 객체의 메모리 주소가 반환된다. 

박싱이 어떻게 이루어지는지 알았으므로, 이제 언박싱에 대해 이야기할 차례다. 다음의 코드에서는 ArrayList에 담겨있는 요소들 중 첫 번째 요소를 가져오려고 하고 있다.
~~~
Point p = (Point) a[0];
~~~
앞의 코드가 작동하려면, 박시오딘 객체 Point 안에 들어있는 필드들을 대상 객체에 복사해야 한다. CLR은 이러한 작업을 두 단계에 걸쳐 수행한다. 첫 번째, 박싱된 Point 타입 내에 들어있는 Point 필드들의 주소를 가져온다. 이것을 **언박싱**이라고 한다. 다음으로, 이 필드들의 값들을 힙 메모리로부터 복사하여 스택 기반의 값 타입 인스턴스 쪽으로 모두 복사해 놓는다.

언박싱이 박싱과 정확히 대칭되는 작업은 아니다. 언박싱 작업은 박싱에 비해 훨씬 적은 연산을 수행한다. 언박싱 자체는 박싱된 객체 안에 들어있는 값 타입(데이터 필드)의 포인터를 얻어오는 것에 불과하다. 박싱된 인스턴스 안에 언박싱된 부분의 포인터를 가져오는 효과가 있다. 그래서 박싱과는 다르게 언박싱 자체는 메모리 복사를 수반하지 않는다. 좀 더 중요한 내용을 강조하자면 언박싱 연산이 이루어진 다음 보통은 필드를 복사하는 작업이 이루어진다는 것이다.

내부적으로 박싱된 값 타입 인스턴스를 언박싱하는 과정을 구체적으로 설명하면 다음과 같다.
1. 만약 어떤 변수에 박싱된 값 타입의 인스턴스의 참조가 null로 설정되어 있다면, NullReferenceException이 발생한다.
2. 참조가 원하는 타입에 대한 박싱 객체를 가리키지 않는 경우, InvaildCastException 예외가 발생하게 된다.
~~~
public static void Main() {
    Int32 x = 5;
    Object o = x;
    Int16 y = (Int16) o;    // InvaildCastException 예외가 발생한다.
}
~~~

### dynamic 기본 타입
C#은 타입 안정성을 중시하는 프로그래밍 언어다. 모든 표현식의 결과가 특정한 타입의 인스턴스로 계산되어 컴파일러가 해당 타입에 대해 제공하는 기능만을 연결해서 일을 처리할 수 있는 것이다. 타입 안정성을 지키는 프로그래밍 언어의 이점은 이를 지키지 않는 프로그래밍 언어에 비해서 개발자들이 컴파일 시점에서 오류를 미리 교정할 수 있으므로 예기치 않은 오류를 더 적게 만날 수 있다는 것이다. 더 나아가서 컴파일 기반의 언어들은 더 크기가 작고 더 빨리 실행되는 코드를 만들 수 있는데, 컴파일 시점에서 미리 가정을 해볼 수 있고 이 중 합당한 것만을 고정된 형태의 IL과 메타데이터로 굳힐 수 있기 때문이다.

리플렉션이나 다른 구성요소와 통신하는 일을 좀 더 쉽게 할 수 있도록, C# 컴파일러는 표현식의 타입을 dynamic 타입으로 만드는 기능을 제공하고 있다. 이 표현식의 결과뿐만 아니라 변수의 타입도 dynamic 타입으로 표시할 수도 있다. 여러분의 코드에서 dynamic으로 표시된 표현식이나 변수상의 멤버를 호출하는 코드를 작성하면, 컴파일러는 특별한 IL 코드를 만들어서 해당되는 작업을 수행할 수 있도록 보조해준다. 이 특별한 코드는 **페이로드**라고 불린다. 실행 시점에서 페이로드 코드는 dynamic 표현식과 변수에 의하여 참조되는 객체의 실제 타입을 기반으로 정확한 연산을 찾아 실행하는 일을 한다.

~~~
internal static class DynamicDemo {
    public static void Main() {
        dynamic value;
        for (Int32 demo = 0; demo < 2; demo++) {
            value = (demo == 0) ? (dynamic) 5 : (dynamic) "A";
            value = value + value;
            M(value);
        }
    }

    private static void M(Int32 n) { Console.WriteLine("M(Int32): " + n); }
    private static void M(String s) { Console.WriteLine("M(String): " + s); }
}

M(Int32): 10
M(String): AA
~~~

어떠한 표현식이라도 dynamic 타입으로 캐스팅할 수 있는데, 모든 표현식의 결과 인스턴스는 Object 타입으로부터 상속받는 타입을 사용하기 때문이다. 보통은 컴파일러가 Object 타입에서 다른 타입으로 암묵적으로 캐스팅하는 코드를 작성할 수 없도록 제한할 것이다. 하지만 컴파일러는 dynamic 타입에서 다른 타입으로 암묵적 타입 변환 문법을 사용하여 코드를 작성하는 것을 허용한다.
~~~
Object o1 = 123;            // 이상없음
Int32 n1 = o1;              // 오류: Object 타입에서 int32 타입으로 암묵적으로 캐스팅 불가
Int32 n2 = (Int32) o1;      // 이상없음: 명시적으로 캐스팅 가능 (언박싱 발생)

dynamic d1 = 123;           // 이상없음: Int32 타입에서 dynamic 타입으로 암묵적 캐스팅 가능(박싱 발생)
Int32 n3 = d1;              // 이상없음: dynamic 타입에서 Int32 타입으로 암묵적 캐스팅 가능(언박싱 발생)
~~~

> **중요**: dynamic과 var 타입을 혼동하지 말기를 바란다. 지역 변수를 var 키워드를 사용하여 선언하는 것은 단지 컴파일러가 표현식에 대한 실제 표현식을 유추하도록 요구하는 단축 표현식에 불과하다. 어디까지나 var 키워드는 메서드 내에서 사용되는 지역 변수를 선언하는데에만 사용할 수 있을 뿐이지만 dynamic 키워드는 지역 변수 외에도 필드, 매개변수에도 사용할 수 있다. 캐스팅을 할 때에도 var 키워드를 사용하여 캐스팅을 할 수는 없지만 dynamic 키워드로는 가능하다. 그리고 var 키워드로 변수를 선언할 때에는 반드시 초기값을 지정해야 하지만 dynamic 키워드에서는 지정하지 않고도 컴파일이 가능하다.

> **중요**: dynamic 타입은 사실 그 자체로 System.Object 타입과 동일한 타입이다. 컴파일러는 표현식에 대해서 여러분이 어떤 코드를 작성했다고 할지라도 일단은 모두 인정하여 전혀 경고나 오류를 일으키지 않는다. 하지만 실행 시점에서 여러분이 작성한 코드가 잘못된 동작을 수행하게될 경우 이에 대한 예외를 일으키게 된다. dynamic 타입이 Object 타입과 동일시된다고는 하더라도 dynamic 타입을 확장하는 확장 메서드를 만들 수 없게 되어 있다. 람다식이나 익명 메서드를 dynamic 메서드 호출을 위한 인자로 지정할 수 없으며 이는 컴파일러가 정확한 타입을 유추할 수 없기 때문이다.

다음의 코드에서는 "Jeffrey Richter"라는 문자열을 포함하는 String 타입의 인스턴스에 대해서 Contains 메서드를 부를 때 문자열 매개변수 "ff"를 호출하고, 실행 후의 Boolean 타입의 결과를 지역 변수 result에 저장하는 리플렉션을 이용하여 지정하고 호출하는 방법을 보여주고 있다.
~~~
Object target = "Jeffrey Richter";
Object arg = "ff";

// 원하는 매개변수 타입과 일치하는 대상 메서드를 검색한다.
Type[] argTypes = new Type[] { arg.GetType() };
MethodInfo method = target.GetType().GetMethod("Contains", argTypes);

// 해당 메서드를 원하는 매개변수를 사용하여 호출한다.
Object arguments = new Object[] { arg };
Boolean result = Convert.ToBoolean(method.Invoke(target, arguments));
~~~

C#의 dynamic 타입을 사용하면, 위의 코드를 다음과 같이 획기적으로 단순화시킬 수 있다.
~~~
dynamic target = "Jeffrey Richter";
dynamic arg = "ff";
Boolean result = target.Contains(arg);
~~~
