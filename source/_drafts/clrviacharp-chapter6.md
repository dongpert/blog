## 서로 다른 종류의 타입 멤버들
* 상수: 상수는 절대불변의 데이터 값을 식별하는 기호다.
* 필드: 필드는 읽기 전용 또는 읽기/쓰기 겸용 데이터 값에 대한 표현이다. 필드는 타입의 상태를 정의하기 위하여 정적 멤버로 선언될 수 있다. 뿐만 아니라 인스턴스의 상태를 정의하기 위한 목적으로 인스턴스 멤버로 선언될 수도 있다.
* 인스턴스 생성자: 인스턴스 생성자는 객체의 인스턴스 필드를 의도하는 대로 초기화하기 위한 목적으로 선언할 수 있는 특별한 메서드다.
* 타입 생성자: 타입 생성자는 타입의 정적 필드들을 의도하는 대로 초기화하기 위한 목적으로 선언할 수 있는 특별한 메서드다.
* 메서드: 멤버로 추가할 수 있는 함수로, 특정한 타입에 대한 상태를 변경하기 위하여 정적 메서드를 선언하거나 특정한 인스턴스에 대한 상태를 변경하기 위하여 인스턴스 메서드로 선언할 수 있다.
* 연산자 오버로드: 연산자 오버로드는 서로 다른 두 객체에 대해 조작하는 방법을 정의하는 메서드다.
* 변환 연산자: 변환 연산자는 암묵적 또는 명시적으로 어떤 객체를 다른 객체 타입으로 변환할 수 있는 방법을 정의하는 메서드다.
* 속성: 속성은 필드와 비슷한 문법을 사용하여 타입 내의 필드를 변경하거나 상태를 조회하기 위한 정적 속성, 객체 내의 필드를 변경하나 상태를 조회하기 위한 인스턴스 속성으로 나눌 수 있다.
* 이벤트: 정적 이벤트와 인스턴스 이벤트는 특정한 타입이 다른 정적 메서드 또는 인스턴스 메서드로 통지를 보낼 수 있는 방법을 제공한다. 이벤트는 보통 타입이나 객체의 상태가 변경될 때 이를 통지받을 수 있도록 할 때 선언하여 기능을 제공할 수 있다.
* 타입: 타입 안에 또 다른 타입을 중첩해서 선언할 수도 있다.
~~~
public sealed class SomeType
{
    // 중첩된 클래스
    private class SomeNestedType { }

    // 상수 필드, 읽기 전용 필드, 그리고 읽기/쓰기가 가능한 정적 필드
    private const Int32 c_SomeConstant = 1;
    private readonly String m_SomeReadOnlyField = "2";
    private static Int32 s_SomeReadWriteField = 3;

    // 타입 생성자
    static SomeType() { }

    // 인스턴스 생성자
    public SomeType(Int32 x) { }
    public SomeType() { }

    // 인스턴스 및 정적 메서드
    private String InstanceMethod() { return null; }
    private static void Main() { }

    // 인스턴스 속성
    public Int32 SomeProp
    {
        get { return 0; }
        set { }
    }

    // 매개변수를 요구하는 인스턴스 속성(인덱서)
    public Int32 this[String s] {
        get { return 0; }
        set { }
    }

    // 인스턴스 이벤트
    public event EventHandler SomeEvent;
}
~~~

## 멤버 접근성
| CLR 용어 | C# 용어 | 설명 |
| :- | :- | :- |
| Private | private | 이 한정자로 선언되 멤버는 멤버가 정의된 타입 내의 다른 메서드나 중첩 타입에서만 접근할 수 있다. |
| Family | protected | 이 한정자로 선언된 멤버는 멤거가 정의된 타입 내의 다른 메서드 나 중첩 타입, 그리고 어셈블리의 내부나 외부에 구분 없이 이 타입을 상속받는 하위 타입에서만 접근할 수 있다. |
| Assembly | internal | 이 한정자로 선언된 멤버는 같은 어셈블리 안에 선언된 타입에서만 접근할 수 있다. |
| Public | public | 이 한정자로 선언된 멤버는 모든 어셈블리와 모든 메서드에서 접근할 수 있다. |

C#에서는 선언하는 멤버에 명시적으로 한정자를 지정하지 않을 경우, 컴파일러는 모든 경우는 아니지만 대개 private 한정자를 자동으로 선택하여 적용하게 된다. 그리고 CLR에서는 인터페이스에 선언되는 모든 멤버들은 반드시 public 한정자만을 사용할 수 있도록 하고 있다.

## 정적 클래스
정적 클래스로 선언하면 컴파일러는 다음의 사항들을 제한한다.
* 정적 클래스는 반드시 System.Object에서만 상속을 받도록 제한된다.
* 정적 클래스는 인터페이스를 구현할 수 없도록 제한되는데 인터페이스 메서드는 반드시 클래스의 인스턴스로 생성될 때에만 호출할 수 있기 때문이다.
* 정적 클래스 안에서는 필드, 메서드, 속성, 이벤트는 모두 정적 멤버로 선언되어야만 한다.
* 정적 클래스는 필드, 메서드 매개변수, 또는 지역 변수의 타입으로 사용될 수 없는데 이러한 경우 모두 클래스를 인스턴스화하는 것과 연관성이 있으므로 이 작업이 허용되지 않기 때문이다.

## 부분 클래스, 부분 구조체, 부분 인터페이스
partial 키워드를 사용하면 한 소스 코드 파일 내에서 여러 조각으로 타입의 선언을 분할하거나 여러 소스 코드 파일에 걸쳐 여러 조각으로 타입의 선언을 분할할 수 있게 하고 나중에 컴파일 단계에서 C# 컴파일러가 이를 하나로 통합하여 컴파일하게 되므로 CLR은 완성된 타입 정의 위에서 작업할 수 있도록 해준다. 왜 이와 같이 여러 조각으로 선언을 분할하는지 그 이유를 살펴보면 대개 다음의 이슈 때문인 경우가 많다
* 소스코드 제어: partial 키워드를 이용하여 해당 타입에 대한 선언을 여러 개의 코드 파일로 분할해 놓는다면, 여러 개발자들이 각자 해당되는 코드 파일을 체크아웃하여 동시에 작업을 진행할 수 있을 것이다.
* 클래스, 구조체, 인터페이스를 단일 파일 내에서 논리적으로 여러 개로 분할하여 선언하기: 구현을 단순하게 만들기 위해서, 단일 소스 코드 파일안에서 필요한 만큼 반복적으로 partial 키워들르 사용하여 구현을 나누는 것을 고려할 수 있다.

## 컴포넌트, 다형성, 버전관리
컴포넌트 소프트웨어 프로그래밍(Component Software Programming, CSP)은 OOP를 이 수준까지 끌어올렸다. 다음은 컴포넌트로서 가져야 할 성격들을 열거한 것이다.
* 하나의 컴포넌트(.NET Framework에서는 하나의 어셈블리)는 "출반(Published)" 된 것으로 간주한다.
* 하나의 컴포넌트에는 식별할 수 있는 아이덴티티(이름, 버전, 문화권, 그리고 공개키)가 존재한다.
* 하나의 컴포넌트에는 영원히 유지되는 고유한 아이덴티티가 존재한다. 어셈블리 안에 들어있는 코드는 다른 어셈블리에 정적으로 링크되는 일이 없으며, .NET 환경에서는 항상 동적 링크만 발생한다.
* 하나의 컴포넌트는 의지하는 다른 컴포넌트들에 대해 정확한 정보를 표시한다.
* 하나의 컴포넌트는 반드시 그 안에 들어있는 클래스와 멤버들에 대해 문서를 만들 수 있어야 한다.
* 하나의 컴포넌트는 반드시 필요로 하는 권한에 대해 자세히 서술하고 있어야 한다. CLR의 코드 엑세스 보안(CAS) 기능을 통해 이를 달성할 수 있다.
* 하나의 컴포넌트는 새로운 서비스를 제공하더라도 변하지 않는 인터페이스나 객체 모델을 제공해야 한다. 서비스 제공이라 함은 원래의 컴포넌트 버전과 호환성을 유지할 수 있는 새로운 버전의 컴포넌트를 제공하는 것이다. 서비스 제공에는 버그 수정, 보안 패치, 그리고 약간의 기능 향상이 포함된다.

.NET Framework에서는 버전 번호가 네 개의 파트로 구분되는데, 메이저 파트, 마이너 파트, 빌드 번호 파트, 리비전 번호 파트로 구성되어 있다. 예를 들어 어셈블리 버전 번호가 1,2,3,4라고 할때, 메이저 파트는 1, 마이너 파트는 2, 빌드 번호 파트는 2, 리비전 번호 파트는 4로 해석할 수 있다. 메이저와 마이너 파트는 보통 어셈블리의 기능상 큰 특징을 설명하기 위한 목적으로 부여되는 것이고, 빌드 번호와 리비전 번호 파트는 어셈블리의 기능 집합 중 일부를 업데이트하는 서비스 제공을 목적으로 부여되는 것이다.

버전 관리 문제는 컴포넌트안에 정의된 타입을 다른 컴포넌트 안에 정의된 타입을 기본 타입으로 사용하려는 경우 발생한다. 분명한 것은 만약 기본 클래스의 버전이 하위 클래스의 버전보다 낮은 버전으로 바뀐다면, 낮은 버전의 클래스의 기능도 같이 변경되어 기능에 문제가 발생하게 될 수 있다.

C#에서는 다섯 가지의 키워드를 통해서 타입 그 자체 또는 타입 멤버의 컴포넌트 버전 관리에 영향을 줄 수 있는 기능을 제공한다.. 이 키워드들은 CLR의 기능과 직접 연결되며 이 기능들은 컴포넌트 버전 관리에 직접 영향을 준다.

| C# 키워드 | 타입에 대한 적용 | 메서드/속성/이벤트에 대한 적용 | 상수/필드에 대한 적용 |
| :- | :- | :- | :- |
| abstract | 여기에 해당하는 타입은 인스턴스로 만들 수 없게 된다. | 이 키워드로 선언된 멤버가 있는 타입을 상속받은 타입이 인스턴스로 생성될 수 있도록 하기 위해서는 반드시 해당 멤버를 구현해야만 한다. | 적용 불가 |
| virtual | 적용 불가 | 이 키워드로 선언된 멤버가 있는 타입을 상속받는 타입에서 이 멤버를 재정의할 수 있다. | 적용 불가 |
| override | 적용 불가 | 기본 타입의 가상 멤버를 현재 타입에서 재정의한다. | 적용 불가 |
| sealed | 여기에 해당하는 타입은 다른 타입의 기본 타입이 될 수 없다 | 이 키워드로 선언된 멤버는 상속 이후에 재정의할 수 없다. 이 키워드는 가상 메서드를 재정의하는 메서드에 대해서만 지정할 수 있다. | 지정 불가 |

### CLR이 가상 메서드, 속성 이벤트를 호출하는 방법
C#을 포함한 대부분의 프로그래밍 언어들은 메서드의 반환 타입을 무시한 채 매개변수 목록의 차이만을 가지고 고유성을 식별하고 있다.

타입을 설계할 때에는 반드시 정의하는 가상 메서드의 수를 가능한 최소화해야 한다. 첫째, 가상 메서드는 비가상 메서드보다 호출 속도가 느리다. 둘째, 가상 메서드는 JIT 컴파일러에 의하여 인라인으로 번역될 수 없어 성능에 영향을 줄 개연성이 크다. 셋째, 가상 메서드는 컴포넌트의 버전 관리를 힘들게 만든다. 넷째, 기본 타입을 선언할 때에는 편의를 위한 오버로드 메서드들의 집합을 제공하는 것이 보편적이다. 만약 이 메서드들이 다형성을 가지게 하려면 가장 복잡하고 핵심적인 메서드에 대해서만 가상 메서드로 정의하고 나머지 모든 메서드들은 비가상 메서드로 남겨두는 것이 좋다. 다른 한편으로, 이러한 가이드라인을 따름으로서 하위 타입에 대한 영향 없이 컴포넌트의 버전관리에 대한 능력을 향상시킬 수 있다.

### 타입의 가시성과 멤버의 접근성을 지능적으로 활용하는 방법
* 버전 관리: 클래스가 처음 만들어지는 시점부터 상속을 봉인하면, 호환성을 무너뜨리지 않고 서도 상속이 가능하도록 봉인을 해제할 수도 있다. 하지만 한 번 클래스가 상속이 가능하도록 설정되고 난 후에는 다시 봉인 상태로 되돌아가서는 안되며 이렇게 될 경우 모든 하위 클래스들에 문제가 발생하게 된다.
* 성능: 가상 메서드에 대한 호출은 비가상 메서드와는 다르게 동작하는데, CLR이 실행 중에 정확히 어떤 타입내에 들어있는 메서드를 호출해야 할 것인지 객체의 타입을 검사함으로 알아내야 하기 때문이다. 클래스가 봉인된 상태이므로 더 이상 하위 클래스가 존재할 수 없음을 확신할 수 있기 때문이다.
* 보안 향상과 예측 용이성: 하나의 클래스는 반드시 자신의 고유한 상태를 지켜야 하고 훼손되지 않도록 보호해야만 한다.

다음은 새로운 클래스를 정의할 때 따르는 스스로의 가이드라인들이다.
* 클래스를 정의할 때에는 만드는 클래스에 확실히 상속을 이용하여 새로운 기능의 확장을 허용하도록 만들기 전까지는 명시적으로 sealed 키워드를 사용 상속이 불가능하도록 만든다. 그리고 또한 클래스를 internal로 설정하는 것을 기본으로 하여 확실히 어셈블리 외부에 공개할 타입으로 결정하기 전짜지는 이를 유지한다.
* 클래스 안에서는 항상 데이터 필드를 private 멤버로 선언하며 이를 어기는 일이 없다. 객체의 상태를 노출하는 것만큼 직접적인 문제를 일으킬 개연성이 높은 일이 없으며, 객체의 상태를 예측 불가능하게 하고, 잠재적인 보안 취약점을 만들기도 한다.
* 클래스 안에서는 항상 메서드, 속성 이벤트까지 모두 private으로 선언하고 비가상 멤버로 선언하는 것을 선호한다. 하지만 멤버를 가상 멤버로서 선언하기로 결정했다면 protected와 internal 멤버로서 선언하는 것을 고려할 수도 있는데 왜냐하면 이러한 경우에는 최대한 많은 기능에 대한 제어권을 부여하고 가능한 상속받는 클래스들이 여기에 의존할 수 있도록 해야 하기 때문이다.
* 알고리즘을 구현할 때 지나치게 복잡해지는 경향이 보인다면, 도우미 타입을 만들어서 각각의 기능들을 여러 조각으로 나누고 이를 캡슐화하려고 시도한다. 그리고 이러한 도우미 타입들을 하나의 복잡한 타입에서 사용할 요량으로 정의를 한다면, 이러한 도우미 타입들을 사용하는 실제 타입 안에 중첩된 타입으로 선언할 것이다. 이렇게 하면, 범위를 한정하는 것이 가능하고 코드를 중첩해서 작성할 수 있으므로 알고리즘을 포함했던 원래의 클래스의 private 멤버에 자유롭게 접근할 수 있을 것이다.

### 타입에 대한 버전 관리와 가상 메서드 사이의 조율
다른 타입의 기본 타입으로 사용하고 있는 특정한 타입에 새로운 멤버를 더하거나 수정할 때에는 조심스럽게 작업해야 한다. 다음의 예제를 살펴보도록 하자.
~~~
namespace CompanyA {
    public class Phone {
        public void Dial() {
            Console.WriteLine("Phone.Dial");
        }
    }
}
~~~
이제 여기에 CompanyB가 BetterPhone이라는 새로운 타입을 정의하였으며, 이 타입은 CompanyA의 Phone 타입을 기본 타입으로 사용한다고 가정해보자.
~~~
numespace CompanyB {
    public class BetterPhone : CompanyA.Phone {
        public void Dial() {
            Console.WriteLine("BetterPhone.Dial");
            EstablishConnection();
            base.Dial();
        }

        protected virtual void EstablishConnection() {
            Console.WriteLine("BetterPhone.EstablishConnection");
        }
    }
}
~~~
CompanyB의 개발자가 이 코드를 컴파일하려고 하면 C# 컴파일러에서는 다음과 같은 경고 메시지가 나타나게 된다.
~~~
컴파일러 경고 CS0108: 'CompanyB.BetterPhone.Dial()'은(는) 상속된 멤버 'CompanyA.Phone.Dial()'를 숨깁니다. 숨기려면 new 키워드를 사용하십시오.
~~~
이 경고를 나타내도록 하지 않기 위해서 BetterPhone 클래스의 Dial 메서드 앞에 new 키워드를 붙일 것을 이야기하기도 한다. 다음은 수정된 BetterPhone 클래스의 코드다.
~~~
numespace CompanyB {
    public class BetterPhone : CompanyA.Phone {
        public new void Dial() {
            Console.WriteLine("BetterPhone.Dial");
            EstablishConnection();
            base.Dial();
        }

        protected virtual void EstablishConnection() {
            Console.WriteLine("BetterPhone.EstablishConnection");
        }
    }
}
~~~
다음은 CompanyB의 개발자들이 작성할 코드의 예시다.
~~~
public sealed class Program {
    public static void Main() {
        CompanyB.BetterPhone phone = new CompanyB.BetterPhone();
        phone.Dial();
    }
}
~~~
이 코드가 실행되면, 다음과 같이 결과가 나타날 것이다.
~~~
BetterPhone.Dial
BetterPhone.EstablishConnection
Phone.Dial
~~~
다른 회사들도 Dial 메서드에서 연결을 성립시키기 위한 기능을 추가하는 것이 매우 유용한 기능이라고 판단하고 있다고 가정하자. 이러한 의견이 CompanyA에 전달되면, Phone 클래스를 다음과 같이 새로 고치게 될것이다.
~~~
namespace CompanyA {
    public class Phone {
        public void Dial() {
            Console.WriteLine("Phone.Dial");
            EstablishConnection();
        }
    }

    protected virtual void EstablishConnection() {
        Console.WriteLine("Phone.EstablishConnection");
    }
}
~~~
CompanyA의 Phone 클래스의 새 버전을 기본 타입으로 하여 CompanyB의 BatterPhone 타입을 새로 컴파일하려고 하면, 컴파일러는 다음과 같은 메세지를 띄울 것이다.
~~~
컴파일러 경고 CS0114: 'CompanyB.BetterPhone.()'EstablishConnection은(는) 상속된 멤버 'CompanyA.Phone.EstablishConnection()'를 숨깁니다. 숨기려면 new 키워드를 사용하십시오.
~~~
CompanyB는 EstablishConnection 메서드에 new 키워드를 지정하여 이러한 의도를 컴파일러에 설명할 수 있다.
~~~
numespace CompanyB {
    public class BetterPhone : CompanyA.Phone {
        public new void Dial() {
            Console.WriteLine("BetterPhone.Dial");
            EstablishConnection();
            base.Dial();
        }

        // new 키워드를 새로 추가하여 기본 타입의 EstablishConnection 메서드와는 무관함을 표시한다.
        protected new virtual void EstablishConnection() {
            Console.WriteLine("BetterPhone.EstablishConnection");
        }
    }
}
~~~
이와 같이 하지 않는 대신, CompanyB는 CompanyA의 새 버전의 Phone 타입이 이제까지 가지고 있었던 Dial 메서드와 EstablishConnection 메서드와 완벽히 의미가 일치한다고 판단하여 기존의 메서드를 지우도록 BetterPhone 메서드를 수정할 수도 있을 것이다. 더 나아가서 CompanyB는 컴파일러에게 EstablishConnection 메서드가 Phone의 EstablishConnection 메서드와 관계가 있음을 설명하기 위하여 new 키워드를 지워야만 한다. 그리고 이것만으로 충분하지 않기 때문에 Phone 타입의 메서드를 재정의하기 위하여 가상 메서드임을 설명하기 위해 붙인 virtual 키워드 대신 기본 타입의 가상 메서드를 재정의하기 위하여 override 키워드로 수정해야 한다.
~~~
numespace CompanyB {
    public class BetterPhone : CompanyA.Phone {
        // 기존의 Dial 메서드를 삭제하고 기본 타입의 Dial 메서드를 상속받았다.

        // new 키워드를 삭제하고 'virtual' 대신 'override' 키워드를 사용하여
        // 기본 타입의 EstablishConnection 메서드를 재정의하고 있음을 설명한다.
        protected override void EstablishConnection() {
            Console.WriteLine("BetterPhone.EstablishConnection");
        }
    }
}
~~~
